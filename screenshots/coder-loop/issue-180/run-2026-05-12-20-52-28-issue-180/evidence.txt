Prior review (#182 comment 4434701972) required acceptance #6 to be executed for real,
not stubbed. This retry rewrites the wire layer to match the exchange protocol and

=== root cause uncovered while attempting #6 ===

The previous run's `channel-heartbeat-service.ts` was incompatible with the actual
`@agent-channel/exchange` wire contract on every layer:

- `/version` returns snake_case `schema_version`; the old service parsed `schemaVersion`
  (camelCase) → always `undefined` → schema-compat probe could never succeed.
- The old probe expected major "1.x"; protocol `SCHEMA_VERSION = "0.1.0"` (pre-1.0).
- `POST /v1/register` requires a full wire envelope with `body.kind = "register.request"`
  and `body.payload.{schema_version, desired_channel_id, capabilities, identity}`;
  the old service POSTed a flat object plus `Authorization: Bearer <token>`.
- Response is also an envelope; old code read `data.channelId` (camelCase, flat) and
  always failed shape validation.
- Capabilities used `channel.list_channels`; protocol enum is
  `[channel.send, channel.receive, discovery.list]`.
- `POST /v1/envelope` (heartbeat) requires `X-Agent-Channel-Sender: <channel_id>`;
  old code sent only `Authorization: Bearer`. Heartbeat body was missing the
  `{ channel_id, sequence }` payload required by `heartbeat.ping`.

Result: unit tests passed (they faked the camelCase / flat-shape response), but every
real register / heartbeat would 400 at the exchange. Acceptance #6 caught this.

=== fix landed in this run ===

`server/services/channel-heartbeat-service.ts` rewritten to speak the real protocol:

- `EXCHANGE_SCHEMA_VERSION = "0.1.0"` exact match (strict, like the exchange).
- `probeExchangeVersion`: GET /version, read `schema_version` (snake_case).
- `registerChannel`: build envelope `{ msg_id, from, to:"exchange/system", ts,
  schema_version, body: { kind:"register.request", payload: { schema_version,
  desired_channel_id, capabilities, identity:{ agent_kind:"fulcrum-client",
  instance_label } } } }`, POST `/v1/register`, parse `body.payload.channel_id /
  registered_at / heartbeat.{interval_seconds,timeout_seconds}`, write back to
  `terminals.channel_id` + `terminals.channel_registered_at`.
- `sendHeartbeatPing`: POST `/v1/envelope` with `X-Agent-Channel-Sender: <channel_id>`,
  envelope `body.kind = "heartbeat.ping"`, payload `{ channel_id, sequence }`.
- `startChannelHeartbeat`: drop the boot-time `cfg.enabled` gate; the timer is always
  running but each tick re-reads fnox and short-circuits if disabled or no channels
  tracked. This means enabling exchange via Settings UI at runtime (without restart)
  now actually starts ticking. Fixes the latent issue caught during e2e setup.
- Capabilities corrected to `[channel.send, channel.receive, discovery.list]`.

Token fnox key is preserved as forward-compat (not transmitted; exchange uses
`X-Agent-Channel-Sender`, not Bearer auth).

`server/services/channel-heartbeat-service.test.ts` rewritten to assert envelope
shape rather than the old flat camelCase shape, so future wire drift is caught
in CI.

=== acceptance #2 soft-deviation decision ===

`drizzle/0081_add_terminal_channel_id.sql` + handwritten `drizzle/meta/_journal.json`
entry are kept as-is for this PR. The migration is two `ALTER TABLE ... ADD COLUMN`
statements; running `mise run db:generate` would not produce materially different
output and risks reshuffling the journal across `main` rebases. Recording this
explicitly in the PR comment as a one-off; future schema changes that touch more
than column adds should go back to the generator.
